# Design Board Prompt Template

Intent:
Create a production-ready light-mode design board for a React shadcn/ui frontend. The board will be used as the visual source of truth for implementation, so make it specific, coherent, and directly buildable.

Product:
<Describe the app, user, domain, and primary workflow.>

Design Thinking:
Understand the product purpose, primary user, tone, constraints, and the one memorable visual idea. Choose a clear aesthetic point of view and execute it with precision. The design can be refined minimal, editorial, dark neon, luxury, playful, brutalist, industrial, soft/pastel, skeuomorphic, or another direction if the user asks for it. Intentionality matters more than intensity.

Always-On Anti-Generic Rules:
Do not generate a generic AI dashboard, Bootstrap/admin-template UI, default blue-gray shadcn screen, predictable equal-card grid, left-nav plus KPI-strip plus plain data-table layout, purple-gradient SaaS hero, or cookie-cutter enterprise screen. Avoid default-looking typography, harsh borders everywhere, timid evenly-distributed palettes, and components that look dropped in without design direction. The UI must feel deliberately designed for this product. If the first composition could be mistaken for a standard admin template, it is a failed design.
Business, operational, or enterprise workflow does not mean old enterprise visual language. Even when the app category is business-facing, keep the UI sparse, spacious, modern, and de-carded. Do not switch to conventional enterprise arrangement just because the product is a business app.
For business and enterprise operational apps, prefer the newly approved modern business language: a slim tonal rail when left navigation is needed, a quiet open top context zone with selective facts instead of a KPI strip, and one main working section with attached controls rather than multiple helper cards.

Style Direction:
If the user provided a strict style, follow it. Otherwise use the skill's default premium minimal reference style in light mode: a designed-object interface with a subtly offset pale background, cool off-white/porcelain content surfaces instead of raw white, a large rounded outer canvas, oversized editorial panels, distinctly rounded or pill-like controls, quiet low-contrast borders, crisp contact-edge elevation only where needed, expressive metric typography, asymmetric composition, carefully stacked/overlapping cards, curated functional data surfaces, generous whitespace, and one or two strong accent colors. The default should feel closer to a premium product design reference than an enterprise admin kit.
Do not default to blue just because the product is a web app. Unless the user specifies a brand color, vary the primary/accent family across fresh generations: deep green, olive, maroon, rust, amber, charcoal, ink-black, teal, forest, cocoa, or other suitable hues are all valid. If the user gives a primary color or palette, build the surrounding palette from it coherently with lighter/darker supporting tones instead of falling back to generic blue-gray UI.

Reference Polish Signals:
Make the board feel polished through the same recurring signals seen in the written reference set: a clear stage/canvas/focal-surface hierarchy, restrained accent use, confident typography with one display or numeric anchor, quiet navigation chrome, consistent pill/radius language, quiet borders, crisp contact-edge lift only where needed, one dominant visual idea, real breathing room, curated data expression, and clean micro-geometry. A merely clean dashboard is not enough.

App-First Rule:
This skill builds app UIs, not marketing pages. Premium and editorial must not mean landing-page hero, portfolio poster, or oversized website H1. Keep the screen operational: navigation, controls, filters, forms, lists, state, actions, and responsive app behavior should be visible and usable. Use expressive type and a strong active object, but keep title scale proportional to app workflow.
Aim for the narrow middle ground between enterprise density and marketing emptiness. The board should feel airy and minimal while still obviously functioning as software.
Keep the distinction between the app UI and the board metadata clear. The app workspace must avoid footer workflow trays and decorative filler, but the board itself must still include a full-width bottom design-system dock because the next step after approval is implementation.
Do not let the hero collapse into a single graph, route, or diagram with no real controls or actions around it. Strong visual objects still need app behavior.
Do not let the hero become a bespoke illustration or infographic that would force the implementation phase to invent the real app later. The approved mock should already read like something that could be built from realistic React plus shadcn/ui surfaces, with normal controls, records, and local actions.
Do not stop at one elegant object plus one primary button. The hero should usually include at least one scoped control, filter, or mode switch and one attached action cluster, adjustment, or decision parameter so it behaves like a working app surface.
Do not solve this by adding a generic lower admin section under the hero. Avoid bolting on a full table, stacked summary cards, note form, or spreadsheet-like support area just to make the app look functional.
Do not solve buildability by laying out a row or grid of same-weight factor cards under the hero. Repeated driver facts should usually be embedded as inline anchors, compact rows, chips, or one concise sidecar instead of becoming a secondary dashboard strip.
Do not solve buildability by reverting to a plain list or table with ordinary helper cards around it. The active object must still feel like one integrated product-shaped surface rather than a standard CRUD page with nicer spacing.
Do not solve utility with a wide bottom decision tray, approval strip, planner-action band, or full-width next-step footer spanning the lower half of the screen. If extra controls are needed, attach them locally to the active object or condense them into one concise nearby notice.
Do not rely on chart strokes, rings, route lines, or decorative analytical marks to create interest while the actual working content stays thin. The reference polish should come from hierarchy, type, spacing, surface tone, and functional grouping instead.

Designed-Object Standard:
The default quality bar is higher than clean SaaS. Do not create one large white dashboard card on an almost-white page and call it premium. The outer stage, app canvas, active object, mobile preview, and token strip need a clear surface hierarchy visible even at thumbnail scale. Use a stronger tinted outer background, a cool off-white app canvas, brighter cool-white focal surfaces, and a few custom product-specific data forms.

Default Composition Requirements:
Use one memorable structural idea. Examples: a large rounded command/work surface, an oversized active-record panel, a layered record stack, a compact icon rail with a secondary navigation tree, a floating workflow timeline, a soft mobile device preview, or an editorial analysis panel. The product-specific structure must drive the composition, not sit as a small widget beside a conventional page. Do not let a table, list, or column-header row dominate the screen unless it has been visually transformed into rich lanes, routes, bands, cards, grouped records, or another domain-shaped object. Make the app screenshot itself distinctive; do not rely on the surrounding design-system strip to make the board look polished.
Avoid the fallback composition of sidebar, top bar, page title, factual stat strip, one or two plain cards, and a bottom button row. Even when sparse, the screen should still have a more intentional structural idea than a cleaned-up admin template.
Avoid solving the page as two or three large neighboring rounded cards of similar size and weight under a header. That still reads like template composition even when spacing and colors are good.
Do not cover the home page in cards just because the product is operational or enterprise. Use card shells only when highlighting one or two important lifted states. Many sections should separate through spacing, tonal shifts, guide lines, or tight graphite contact-edge elevation instead of full bordered cards.
For business apps specifically, avoid the pattern of open top zone plus side summary card plus lower settings card. Keep the top zone open, keep the main working section singular, and let the sparse structure itself create the polish.
Avoid turning key facts into a long thin summary strip under the title. Integrate the most important facts into the active object, one supporting module, or a richer record band so the screen feels composed rather than assembled from sections.
Do not replace the KPI strip with a softer fact strip. A neat row of equal-weight facts above the active object still reads like dashboard scaffolding even when the borders are light.
Do not explain a route or workflow with stage labels or numbered steps across the top. A route with a step header is still a process grid.
Avoid repeated icon-label-value columns in a flat metadata rail. That pattern reads like generated admin UI. Group facts more selectively and with more visual hierarchy.
Avoid a top horizontal metric/KPI strip as the decision summary. It makes the mock feel like a generic dashboard even when the rest is polished.
Include at least one product-specific visual data object or structured surface that is richer than plain text plus buttons: for example a compact weighted stack, segmented risk bar, approval path object, composition block, timeline cluster, resource strip, health band, or other domain-specific form. Keep it sparse, but make it the primary way the user understands the decision. It should still look buildable from normal app primitives rather than like custom poster art. If it feels optional or decorative while a normal list/table carries the real workflow, the design failed.
Do not fake an integrated object by connecting separate row cards to a central price, status, or score card. If the structure is made from multiple similar feeder cards, it is still a card/list composition. Instead, embed the facts inside one continuous object: a band, curve, field, stack, map, lane, or other custom surface.
At least one major region should be open, embedded, or only partially bounded instead of a full bordered card. Use bands, subtle separators, guide lines, typography zones, or canvas-integrated sections so the layout feels designed rather than boxed. If possible, keep the page mostly de-carded and use only one or two lifted cards at most for emphasis.
For CPQ, ERP/DRM, and similar business tools, a strong default is: slim rail, open context header, one commercial/operational levers section, compact local actions, and a calm mobile companion. Prefer this over a dashboard stack of summary cards.
Make the focal object visibly more special than its neighbors. Supporting modules should usually be quieter, less boxed, or lower contrast than the main surface.

App Scale:
Use app-scale display typography. Headings can be distinctive and larger than utility labels, but they should not consume the viewport like a website hero. Keep enough above-the-fold room for controls, records, actions, status, and analysis modules.
Use typography to create polish through contrast: one confident heading or numeric anchor, then smaller lighter utility text around it. Do not make every label the same weight or size.

Spacing And Density:
Default to spacious minimal composition. Use large gutters between regions, generous padding inside panels, fewer visible records, fewer simultaneous modules, and clear blank space around the main object. Let important elements breathe. Avoid filling every available area with controls, rows, metrics, charts, or annotations.
Do not try to prove usefulness by filling every quiet area. Preserve blank regions and calm bands as part of the design.

Progressive Disclosure:
The first screen does not need to show the whole product. Show the user's primary task, a small amount of decision-critical context, and clear paths to secondary detail through tabs, scroll, drawers, sheets, drill-down pages, or secondary routes. Data-heavy does not mean visually dense. A calm first page with fewer visible objects and clear next actions is better than a crowded page full of widgets.

Minimum Necessary Content:
Every visible module must earn its place by helping the immediate task. Do not show secondary lists, timelines, version history, audit activity, detailed routes, analytics, or multiple parallel panels just because they exist in the product. Collapse, summarize, or route them away until the user asks for that detail. Prefer one strong active workspace with a small amount of supporting context over a page that exposes all adjacent product areas.
If in doubt, remove one whole supporting module instead of shrinking everything to fit. The first screen should feel intentionally incomplete in a good way: calm, focused, and ready for drill-down.
Lifted notice/action cards must stay concise. Do not turn them into stacked mini-pages with several titled subsections, long explanatory copy, and many equal-weight buttons. Use one clear issue, short support copy, one primary action, and only the minimum quiet secondary actions needed.

Local Spacing:
Do not let secondary modules become cramped. Side analysis stacks, data/composition bars, workflow routes, steppers, and bottom action groups need their own breathing room. Use larger vertical gaps between modules, more row height in bar groups, enough room between labels and values, and a clear gap above action buttons. On mobile, increase spacing further and show fewer simultaneous details so the phone view feels calm and usable.

Mobile Prioritization:
Do not squeeze every desktop module into the mobile preview. Mobile may omit, summarize, or collapse secondary desktop information. Prefer a calm mobile flow with only the information needed for the immediate action plus clear primary actions. Use larger vertical gaps and touch-friendly controls. If including every desktop section makes the phone cramped, remove the lower-priority section.

Background And Surface Contrast:
Use a clear but tasteful offset between the outer page/background and actual content surfaces so the product UI pops. The outer background should be visibly tinted and slightly darker than the main content: cool gray, glacier gray, porcelain gray, mist, or faint blue-gray. The main app canvas should be lighter cool off-white, and focal elevated surfaces may be brighter cool white/off-white. Avoid pure white everywhere. Use tonal contrast, selective light borders, and crisp contact-edge shadows only on selected lifted elements to separate surfaces. The tinted outer stage must remain visible on desktop as well as mobile; keep enough margin around the app frame so the desktop view does not become one flat pale surface.
Navigation chrome should have a little more contrast than the content canvas so the workspace structure reads immediately. Sidebars, rails, top bars, tabs, or command headers should be slightly cooler, darker, or more tinted than the main canvas without turning into heavy dark slabs. Good default signals are soft graphite, stone, or a very light muted shade of a darker primary-family color.
For light-mode default boards, a sidebar should usually be tonal and quiet: porcelain, mist, stone, warm gray, soft graphite, muted brand tint, or a low-contrast primary-derived side surface. Do not default to a charcoal or black sidebar slab unless the user explicitly requests a high-contrast style or the product's brand makes it necessary.
Do not add scenic or decorative background art behind the workspace. No leaves, mountain scenes, abstract lifestyle illustrations, or marketing-style ambient imagery unless the product explicitly calls for it. If atmosphere is needed, keep it structural and tonal only.

Navigation Contrast:
If the product uses a sidebar, icon rail, secondary nav, top nav, tabs, command bar, breadcrumb, or filter bar, make that navigation chrome readable and intentionally anchored through subtle tonal separation. Navigation should be slightly cooler, slightly darker, or slightly more tinted than the app canvas, but never a saturated or heavy colored block that overpowers the workspace. In light mode, prefer a soft graphite, stone, or very light muted shade of a darker accent family over same-white chrome. Use a distinct surface token, low-contrast divider, inset edge, active pill, and tight crisp grey contact edge where useful. Sidebar/rail shadows may be used, but they must be a 1-2px grey contact edge close to the surface; avoid feathered blur, broad vertical clouds, or harsh strips. Active nav states should be stronger than inactive ones without making the whole nav high contrast. Mobile navigation affordances such as hamburger buttons, bottom nav, tab bars, and drawers should carry the same subtle surface contrast and contact-edge elevation when they appear. Do not force a sidebar or top nav into every app; apply this only to the navigation pattern that fits the product.
Choose the navigation pattern that fits the product instead of defaulting to a sidebar. Some apps want a slim rail or sidebar, but others should use a top navigation bar, top tabs, split header, command surface, or bottom navigation. If a left navigation aid is needed but a full menu is not doing much work, prefer a slim rail before a labeled sidebar. If a sidebar is not pulling real weight for the workflow, do not include it just because it is common in component libraries.

Sidebar Drift Rule:
When a sidebar is appropriate or explicitly requested, treat it as quiet navigation chrome only. Adding a sidebar must not make the main workspace regress into a generic enterprise admin layout. Design the main canvas as if it had to clear the same quality bar without the sidebar; the sidebar is just wrapped around that stronger workspace. The main content still needs the same modern, clean, minimal, spacious, airy composition expected from no-sidebar mocks: one domain-shaped active object, broad gutters, open or banded regions, selective cards, few visible rows, and a clear current decision. Reject the design if the sidebar is followed by the usual enterprise pattern of title, filters, KPI strip, dense row section, right card, and bottom actions. CPQ, ERP, and similar operational products may use sidebars, but their main canvas should stay premium, de-carded, and product-specific. In light mode, avoid dark sidebar slabs by default; they often signal enterprise admin even when the main content is decent.

Action Placement Rule:
Primary actions should feel attached to the active object or current decision. Use a concise lifted notice, an inline action near the selected object, or a small floating control. Avoid detached bottom metadata/action rails that span the page like a form footer, especially in sidebar apps, because they pull the composition back toward enterprise software.
Avoid full-width recommendation bands under the active object. Even when the styling is light, they behave like footer rails and flatten the hierarchy.
Do not split coaching or readiness screens into separate top summary modules for readiness, drivers, training load, and session context. Those facts should merge into one recommendation surface.

Radius System:
Use a coherent radius scale across the board. Pick 3-4 radius tokens and apply them consistently: largest for outer app frame/shells, medium-large for panels/cards, medium for rows/inputs/buttons/tabs, and small or pill for chips/badges/icon buttons. Repeated cards in the same group must share radius, border strength, and shadow depth. Buttons and inputs in the same control family should not have unrelated corner shapes. The visible UI must match the radius tokens shown in the design-system strip.
Avoid the middling enterprise radius that makes buttons and controls feel like a default admin kit. In the default premium direction, prefer a rounder more obviously modern control language and keep it consistent across buttons, tabs, segmented controls, and smaller action surfaces. Visible 4px/6px/8px-style control radii usually still read administrative; push the rounded system further or go intentionally sharp. Use zero or near-zero radius only when a clearly intentional sharp style is being pursued.

Shadow And Border Treatment:
Avoid old-school card shadows. Do not use large, dark, hard-edged, blurry, feathered, ambient, or dirty gray drop shadows. The only acceptable light-mode shadow is a crisp narrow grey contact edge, similar to a premium selected nav item or tiny popover edge. It should look like a 1-2px grey lifted edge, not a soft shadow. Use this token family only:
- Contact 0: no shadow, only a 1px low-opacity border.
- Contact 1: `0 0 0 1px rgba(16,24,40,0.07), 0 1px 0 rgba(16,24,40,0.08)`.
- Contact 2: `0 0 0 1px rgba(16,24,40,0.08), 0 1px 2px rgba(16,24,40,0.10)`.
- Contact 3 for popovers only: `0 0 0 1px rgba(16,24,40,0.08), 0 2px 3px rgba(16,24,40,0.10)`.
No blur radius above 3px. No y-offset above 2px. No shadow spread. No visible aura around the component. No generic `shadow-sm`, `shadow-md`, Bootstrap cards, gray glow, soft ambient depth, or feathered shadow. Large panels should separate through surface tone, radius, spacing, and a low-contrast border; they should usually use Contact 0, not visible shadow. Use Contact 1-3 only on selected nav, selected records, popovers, dropdowns, floating controls, mobile devices, and one or two focal surfaces. If the shadow is visible as a soft cloud outside the edge, the design fails. If the contact edge cannot be rendered as a tight 1-2px grey line, use no shadow instead of a soft shadow. Design-system elevation swatches should also demonstrate tight contact edges; do not draw the token examples as exaggerated floating cards.

Elevation Usage:
The design-system strip may show elevation tokens, but those tokens are not decorative samples. The main UI must actually use the contact-edge tokens on meaningful lifted states: navigation chrome or active nav, a selected/active record lane, an important notice/card, compact floating controls, popovers/sheets, mobile navigation affordances, mobile device preview, or one important focal surface. Important notice/cards should not rely only on tinted border color; combine a brighter focal surface, Contact 1 or Contact 2, and clear spacing so the lift is visible. The depth should be visible at thumbnail scale as a crisp lifted edge, but should never look like a soft card shadow. Do not use elevation on every card, and do not let shadows become the primary separator for large panels. If the screen is separated almost entirely by borders while the dock shows elevation tokens, or if mobile controls are flat while the dock advertises elevation, the design failed.

Border Discipline:
Keep borders soft, quiet, and mostly neutral in light mode. Do not outline major cards, active objects, maps, or notices with dark or heavy colored strokes. Avoid validation-style framing where a full card is wrapped in a colored outline. Status should appear through soft tint fills, small pills, icons, short accent rails, corner accents, or inline labels. In light mode, border emphasis should stay soft and pastel-leaning; if a colored border is one of the darkest or sharpest things on the screen, the design fails.
Do not use borders as the main way to define every region. The refined look should come first from surface tone, spacing, grouping, and hierarchy.

Open Composition Rule:
Do not box every functional group. At least one major secondary region should be an open area on the app canvas using typography, spacing, pills, bars, dividers, or background bands instead of a full card shell. Use fewer cards, larger gaps, and stronger hierarchy. The board should look designed, not assembled from a component library.
If the primary screen still feels like a row of equal cards after the first pass, redesign it with one dominant open or embedded workspace and more subordinate supporting surfaces.

De-Carding Rule:
The active hero area should not look like a standard white card with a drop shadow. Prefer an open hero band, a faintly tinted panel, a partial-border surface, or content placed directly on the app canvas. If the layout needs cards, reserve visible card treatment for the selected row, a floating control, or a mobile preview, not every region.

Accent Rule:
Use strong accent color as a badge, short rail, selected tab, small icon, partial edge, or focused metric. Do not wrap large hero regions in a full bright outline unless the user explicitly asks for an alert-heavy visual system; a full coral rectangle around the main panel usually reads like form validation, not premium UI.

List Rule:
Do not render important records as a bordered table box, process grid, option matrix, or column-header row layout. Use a sparse set of rich lanes with generous height, open sibling rows, a clear selected/lifted state when useful, sparse columns, and clear whitespace. If the records read as a spreadsheet, procurement table, exercise table, ticket list, shipment grid, or admin table, redesign them around a domain-specific object.
Do not rebuild table density inside the focal lane by adding many tiny columns, repeated metadata cells, column headers, numeric cells, or per-row utility clutter. Keep the visible rows few, curated, and visually calm.
Avoid photo-heavy row thumbnails unless the product genuinely needs imagery. In most app workspaces, thumbnails add noise and reduce the reference-like minimal quality.

Domain Object Over Rows:
The main workspace must read as one integrated product-specific object, not a decorated list. A stack of route rows with icons is still a list. A stage-by-stage route/process grid is still a table. Three neighboring cards or columns are still a board. Use rows, cards, grids, and columns only as subordinate details after the primary object is established. The primary object can be a pressure curve, route map, replenishment band, quote composition object, blocker cluster, active thread, session path, readiness ring, resource map, or another workflow surface specific to the app. If the board would still work after replacing the domain content with generic CRM records, redesign it.
For business-facing operational tools, one dominant operating surface made from three or four rich adjustable rows or levers is acceptable when those rows clearly behave as one calm working section rather than a spreadsheet. In that mode, keep the rows tall, sparse, and attached to shared context: no column headers, no dense metadata cells, no equal helper cards around them, and no table-like striping.
Do not fake an integrated object by placing a central circle, price box, score panel, or status node between two mirrored row stacks or feeder cards. That is a split table. For CPQ, never draw material, labor, freight, overhead, discount, surcharge, or approval facts as separate row cards feeding a central quote card; make them segments, bends, ticks, embedded labels, or small anchors inside one continuous margin-pressure field, curve, waterfall, or stacked band. For fitness, do not use a left score panel plus center recommendation card plus right signal-list card plus bottom coach-note strip; merge readiness drivers, session path, and coach guidance into one recommendation surface with only sparse inline annotations.
Do not hide a dashboard/process layout inside one large wrapper. A big bordered shell split into labeled columns, dividers, or mini-modules still fails. For CPQ, a five-part sectioned board for build-up, adjustments, margin, risk, and outcome still reads like staged enterprise UI. For ticketing, do not use a normal alert card for the blocker plus a second next-action card beneath it; the blocker must feel embedded in the dependency field. For fitness, a single large panel that still splits into readiness, signals, load, recommendation, and session sections remains a dashboard and fails.
For CPQ and ERP, do not place the real working decision in a wide bottom tray under the hero. A lower band with approval steps, planner choices, multi-column actions, or a footer CTA row still reads like enterprise workflow furniture even when the main object is better.
For fitness/coaching, a better default is an open recommendation band or path running across the canvas. Readiness, load, signals, and the chosen session should read as one flow with inline markers, not as titled compartments inside a bordered overview card.
Do not let minimal composition strip away usability. The main object must still include concrete product facts, state, owner/context, and at least one direct action; avoid turning the hero into a mostly decorative chart or line with too little operational content. In most cases it should also expose one local control, filter, or mode switch so the workspace feels interactive, not merely annotated. Add that utility close to the active object rather than as a detached lower-half admin section. Navigation chrome should remain slightly darker or more tinted than the content canvas so the app frame stays readable.
For ticketing, do not render the active blocker as a normal rectangular card with a tinted outline and then attach smaller issue cards to it. The selected blocker should feel embedded in the dependency field itself through a soft zone, capsule cluster, or integrated focal surface rather than a standard card shell.
Avoid both wide horizontal blocker cards on desktop and tall phone-sized blocker cards on mobile. If the blocker reads like a standard alert box with a perimeter outline, header, metadata rows, and one CTA button, the design failed.

Also check whether it feels like a website/marketing hero instead of an app workspace. In light mode, reject a default black/charcoal sidebar slab unless explicitly requested; use a softer tonal sidebar instead. Reject left navigation that grows into a wide labeled enterprise sidebar by habit; for this skill's default business-app baseline, left navigation should usually stay as a slim rail or a narrow quiet sidebar. Check whether the main object is actually integrated, or whether it is only a vertical stack of decorated rows, a stage-by-stage process grid, route options, tickets, exercises, shipments, a central circle/price/score object between mirrored row stacks, or three equal cards; if it is mostly rows, feeder cards, grids, mirrored lists, or equal cards, redesign around one larger domain object. Design System:
Use a shadcn/ui-compatible component system with CSS variable theming and Tailwind implementation, but do not make it look like default shadcn. In the bottom dock, show a clear light-mode color palette, foundation surface swatches for stage/canvas/navigation chrome, border strength, border radius, Contact 0-3 crisp grey contact-edge tokens, status colors, muted text, focus/selection states, typography scale, and a few build-relevant component/state examples such as buttons, fields, chips, tabs, menus, rows, or notices. Label the elevation samples explicitly as Contact 0, Contact 1, Contact 2, and Contact 3; do not invent a generic elevation scale or soft 4px/8px/12px shadow tokens. Use the exact values: Contact 0 is no shadow plus a 1px border; Contact 1 is `0 0 0 1px rgba(16,24,40,0.07), 0 1px 0 rgba(16,24,40,0.08)`; Contact 2 is `0 0 0 1px rgba(16,24,40,0.08), 0 1px 2px rgba(16,24,40,0.10)`; Contact 3 is `0 0 0 1px rgba(16,24,40,0.08), 0 2px 3px rgba(16,24,40,0.10)`. The Contact area must look like neutral elevation metadata: blank or off-white sample surfaces plus text. Do not render Contact tokens as colored chips, palette swatches, semantic badges, avatars, or role-color blocks. For the default rounded system, keep the dock's radius samples in a clearly modern family such as 12px, 16px, 20px, 24px, and full pill; do not drift back to 4px/6px/8px admin radii unless you are explicitly exploring a sharp system, in which case show zero radius intentionally. The dock should be readable enough to guide later theming work, and it must stay present even when the main app screen becomes cleaner and more minimal. The implementation will later generate a separate dark-mode companion board from this approved light board, so do not make the first board dark.

Implementation Feasibility:
Use components that can be built with React, shadcn/ui, Tailwind, lucide icons, and local mock data. Favor structures that map cleanly to tabs, cards, tables, charts, lists, dialogs, sheets, popovers, badges, buttons, forms, and lightweight custom layout primitives. Avoid focal compositions that depend on heavy bespoke graphics, poster-like illustration, or complex vector art unless essential to the product.

Constraints:
No generic landing-page hero unless the product is specifically a landing page. No poster-like website hero for app products. No placeholder lorem ipsum. No vague decorative filler. No illegible dense microtext. No random gradients or purely atmospheric imagery. No bland enterprise dashboard defaults. No small-radius white cards arranged in a mechanical grid. No plain spreadsheet table as the main visual answer. No hero chart, hero ring, or hero route illustration replacing actual functional content. No crowded dashboard canvas. No visibly broken timeline/stepper/row/card geometry. No old-school heavy drop shadows, no Bootstrap shadows, and no shadows other than Contact 0-3 crisp grey contact-edge tokens. Do not omit the bottom design-system dock. The result should look like a real app that shipped and has a memorable visual identity.
